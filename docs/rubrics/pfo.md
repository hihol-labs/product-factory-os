# Product Factory OS Rubric

Use this rubric when reviewing the methodology itself or a project generated by Product Factory OS.

## Critical

| ID | Check | Pass Criteria |
|---|---|---|
| PFO1 | Classification confidence | Product type, domain, complexity, modules, infrastructure, and ambiguity/confidence are documented. |
| PFO2 | Template compliance | Selected modules match the product template or deviations are justified. |
| PFO3 | Compiler artifacts | `PRODUCT_BLUEPRINT.md`, `BUILD_PLAN.md`, and `EXECUTION_GRAPH.md` exist and agree. |
| PFO4 | Execution graph validity | Nodes, transitions, checkpoints, and repair paths validate with `scripts/validate_execution_graph.py`. |
| PFO5 | State consistency | `.codex-memory/STATE.json` matches current stage, current node, artifacts, gates, and next action. |
| PFO6 | Gate traceability | Review, tests, security, dependency, hardening, and deploy-readiness gate results are recorded. |
| PFO7 | Resume safety | A new Codex session can resume from memory without re-discovering core context. |
| PFO8 | Unit context safety | Autonomous or delegated execution has `.pfo/UNIT_CONTEXT_MANIFEST.json` with inputs, write scope, gates, and verification. |
| PFO9 | Verification recovery | Missing or ambiguous verification evidence records `RECOVERY_REQUIRED` and a repair path instead of passing. |

## Important

| ID | Check | Pass Criteria |
|---|---|---|
| I1 | Strategy fit | Strategy artifacts exist for non-trivial products. |
| I2 | Module ownership | Each execution node has owner role, inputs, outputs, and verification. |
| I3 | Drift control | Docs, code, state, and execution graph are updated together. |
| I4 | Voice-first continuity | Short natural-language commands still route into the same state machine. |
| I5 | Learnings extraction | Significant milestones or repairs record durable decisions, lessons, patterns, or surprises. |
